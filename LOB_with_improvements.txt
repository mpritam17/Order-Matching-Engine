Title: Low-Latency Order Matching Engine: Implementation Plan
Author: Pritam

Project Overview
- Development of a high-performance, low-latency order matching engine in C++.
- Core focus: nanosecond-level execution, deterministic latency, lock-free data structures, and zero-allocation critical paths.

Weekly Work Plan

Week 1: Core Data Structures & Architecture
- To Do: Design the Limit Order Book (LOB). Implement the price level and order node structures.
- Implementation: Use intrusive doubly-linked lists for order tracking at each price level.
- Implementation: Use a flat array or a custom Robin Hood hash map for mapping order IDs to node pointers.
- Focus: Cache locality. Ensure structures fit within standard L1/L2 cache lines (typically 64 bytes).

Week 2: The Matching Algorithm
- To Do: Implement Price-Time Priority matching logic.
- Implementation: Build functions for Add, Cancel, and Modify orders.
- Requirement: Ensure Add and Cancel operate in O(1) time complexity.
- Requirement: Handle aggressive orders (market orders or marketable limit orders) crossing the spread.
- Focus: Algorithmic efficiency and minimizing branch mispredictions during the matching loop.

Week 3: Concurrency & Lock-Free Mechanics
- To Do: Implement communication channels between network/ingress thread and matching engine thread.
- Implementation: Build a Single-Producer Single-Consumer (SPSC) lock-free ring buffer.
- Implementation: Use std::atomic and explicit memory ordering (release/acquire).
- Focus: Avoid mutexes and locks; ensure thread pinning and core isolation.

Week 4: Memory Management
- To Do: Eliminate runtime heap allocations in the critical path.
- Implementation: Create custom memory pools and object arenas for order nodes.
- Implementation: Preallocate all required memory at startup.
- Focus: Determinism (avoid latency spikes from new/malloc during live trading).

Week 5: Networking & I/O
- To Do: Build ingress (order parsing) and egress (market data dissemination) components.
- Implementation: Use epoll or io_uring for efficient I/O.
- Implementation: Parse incoming FIX or binary protocols into internal structs.
- Focus: Kernel bypass concepts and zero-copy handling.

Week 6: Benchmarking, Profiling & Tuning
- To Do: Stress test and measure latency percentiles.
- Implementation: Use Google Benchmark for microbenchmarks.
- Implementation: Build replay mechanism with historical tick data.
- Measurement: 50th, 90th, 99th, and 99.9th percentile latencies with RDTSC.
- Focus: Identify cache misses with perf and tune compiler flags (-O3, -march=native, LTO).

============================================================
2) Recommended Improvements
============================================================

A. Concurrency Model (Highest Impact)
1. Strict single-writer matching core
- Use one dedicated matching thread per symbol (or symbol shard).
- All LOB mutations happen on that thread only.
- Other threads communicate via lock-free queues.
- Benefit: deterministic behavior and no lock contention in matching path.

2. Staged queue topology (not only one SPSC)
- Keep SPSC between adjacent stages where possible.
- If multiple ingress producers exist, use MPSC only at boundaries, then handoff to SPSC into matcher.
- Candidate queue libraries: boost::lockfree::spsc_queue, folly::ProducerConsumerQueue, moodycamel::ReaderWriterQueue.
- Benefit: scales ingress without sacrificing matcher determinism.

3. Explicit memory-ordering contract
- Producer: write payload then publish index with release.
- Consumer: read index with acquire then consume payload.
- Document ordering assumptions near queue code and assert ownership invariants.
- Benefit: avoids subtle lock-free correctness bugs.

4. Sequencer before matcher
- Assign monotonic sequence numbers to inbound events.
- Persist event log (WAL) for replay and recovery.
- Benefit: deterministic replay, easier debugging, better auditability.

5. Bounded backpressure policy
- Define behavior when queue is full: reject, drop, or throttle.
- Track queue depth, enqueue failures, and lag per stage.
- Benefit: graceful degradation under burst load.

B. Data Structures and Memory
1. Dense price indexing when possible
- For bounded tick ranges, use direct-index arrays for price levels.
- For wide ranges, use intrusive tree/map plus top-of-book cache.
- Benefit: predictable latency and better cache locality.

2. Faster order ID lookup containers
- Prefer flat hash maps (e.g., absl::flat_hash_map, ankerl::unordered_dense) over node-based containers.
- Benefit: fewer cache misses than typical node-heavy maps.

3. Preallocate all hot-path objects
- Preallocate order nodes, queue slots, parser messages, cancel/replace envelopes, and market data events.
- Benefit: removes allocator jitter from live path.

C. OS/Hardware-Level Tuning
1. Pin threads and enforce NUMA locality
- Pin matcher/ingress/egress to dedicated isolated cores.
- Keep memory pools NUMA-local to thread affinity.
- Benefit: reduced migration and remote-memory penalties.

2. Huge pages for arenas/rings
- Back hot arenas and ring buffers with huge pages where practical.
- Benefit: lower TLB pressure.

3. Network stack evolution path
- Start with epoll + tuning.
- Move to io_uring if syscall overhead matters.
- Evaluate AF_XDP/DPDK for extreme latency targets.
- Benefit: staged complexity while preserving roadmap to lower latency.

D. Correctness and Performance Validation
1. Deterministic replay harness
- Re-run recorded order flow and compare book/trade outputs exactly.

2. Property-based invariants
- No negative quantity.
- Strict price-time priority.
- Quantity conservation checks.
- Idempotent cancel behavior.

3. Tail-latency measurement discipline
- Report p50/p90/p99/p99.9 and max under sustained and burst traffic.
- Watch for coordinated omission in load generation.

E. Practical C++ Stack Suggestions
- Queue: boost::lockfree / folly / moodycamel
- Hash map: absl::flat_hash_map or ankerl::unordered_dense
- Memory: custom arenas/slabs and optionally std::pmr for non-critical paths
- Benchmarking: Google Benchmark + replay driver
- Profiling: perf (and VTune/eBPF if available)

3) Proposed Next Build Blueprint (Actionable)
============================================================

1. Build pipeline as stages
- Stage 1: Ingress thread parses wire message into preallocated internal event.
- Stage 2: Sequencer assigns global/order-stream sequence.
- Stage 3: Matcher thread consumes and mutates LOB (single writer).
- Stage 4: Egress thread publishes fills/book updates.

2. Keep matching core lock-free by ownership, not shared mutation
- No mutex in matching path.
- No shared write access to order book structures.

3. Add runtime observability counters
- Queue depth high-water mark.
- Event lag (ingress->match and match->egress).
- Reject/drop count.
- p99.9 and max latency per message class.

4. Freeze critical-path allocations
- Validate no allocations in add/cancel/modify/match loops using allocator hooks or sampling checks.
